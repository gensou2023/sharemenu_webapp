## スライド 3: 対象プロダクトの概要

**プロダクト名**: MenuCraft AI（シェアメニュークラフト）
**概要**: 飲食店オーナー向け AI メニュー画像自動生成 SaaS

**主要機能**:
- チャット形式でメニュー情報を入力
- AI（Google Gemini 2.0 Flash）が5サイズの画像を自動生成
- 管理画面（プロンプト管理・ユーザー管理・APIログ等）

**技術構成**:
- Next.js 15 / React 19 / TypeScript
- Supabase（PostgreSQL + Storage）
- Vercel（ホスティング）

**本番URL**: https://sharemenu-webapp.vercel.app

**ノート**: ターゲットは30〜60歳の飲食店オーナー。デザインスキルがなくてもSNS用のメニュー画像を3分で作れることを目指したプロダクト。技術的にはモダンなフルスタック構成を採用している。

## スライド 4: 開発手法 — エージェントロール制

**手法**: Claude Code 上に擬似的なチーム体制を構築

| ロール | 担当 |
|--------|------|
| PdO（プロダクトオーナー） | 要件定義・優先度判断 |
| デザイナー | デザインシステム・UIパターン |
| フロントエンド | React/Next.js 実装 |
| バックエンド | API・DB・認証 |
| QA | テスト・品質保証 |
| UX | ユーザー体験設計 |

**ポイント**: 各ロールの振る舞いを Markdown ファイル（`docs/agents/*.md`）で定義し、AIがロールに沿って判断・実装するようガイドした。

**ノート**: 1人の開発者がAIエージェントに対して「今はデザイナーとして判断してほしい」「バックエンドエンジニアとして実装してほしい」と切り替えながら開発を進める形式。ロール定義ファイルにはペルソナ・判断基準・成功指標を記述している。

## スライド 5: 開発手法 — Issue 駆動 + Sprint 制

**開発フロー**:
```
企画テキスト → /plan-issues で Issue 一括作成
  → /priority-review で4軸評価・Tier分け
    → Sprint 単位で実装 → PR → develop → main
```

**4軸評価**:
1. ユーザー価値（離脱率への影響）
2. 収益貢献（Pro転換への寄与）
3. 技術リスク（負債の蓄積度）
4. 実装コスト（S / M / L / XL）

**ブランチ戦略**: main ← develop ← feature/* / design/* / fix/*

**ノート**: 企画から実装までのプロセスを「スキル」として定型化している点が特徴。/plan-issues を実行すれば、企画テキストから依存関係付きのGitHub Issueが一括で作成される。優先度の判断も属人化させず再現可能なフレームワークに落とし込んだ。

## スライド 7: タスク別の効率比較

| タスク | 従来見積 | Claude Code |

| DB統合 + 管理画面 + デプロイ（2,300行） | 7〜10営業日 | 半日 |

**効率化率**: おおむね 5〜10 倍

**ノート**: 特に効率的だったのはボイラープレートの生成とパターンの横展開。LP用に確立したデザインパターンを、ダッシュボード・チャット画面に「強度を変えて」一括適用する作業は、人手では丸1日かかるがClaude Codeでは1.5時間で完了した。

## スライド 8: 効率化に寄与した要因

**1. 一気通貫の実装能力**
- 企画書作成 → DB設計 → API実装 → UI構築 → デプロイまでを1セッションで連続実行
- 文脈の切り替えコストがゼロ

**2. パターンの横展開**
- デザインシステム（カラートークン・ホバーエフェクト等）を1画面で確立 → 全画面に統一適用
- 「強度レベル」（LP > Dashboard > Chat）を設計し段階的に展開

**3. ドキュメントの同時生成**
- 実装と並行して作業記録・調査レポート・セッションログを自動生成
- 開発後に「ドキュメントを書く」工程が不要

**4. デバッグの自律実行**
- Chrome MCP 連携でDOM計算値を解析し、CSSバグの根本原因を自動特定

**ノート**: 従来の開発では「設計→実装→テスト→ドキュメント」が分断されがちだが、Claude Codeではこれらが1つの連続したプロセスになる。特にドキュメントが副産物として自動生成される点は、長期的なプロジェクト運用において大きな利点。

## スライド 9: 発生した課題 — セッション跨ぎインシデント

**事象（2026-02-13）**:
- セッション引き継ぎ時、前セッションのコンテキストに含まれていた指示書の内容を「実際に読んだファイル」と誤認
- **存在しないファイルに基づいて実装を開始** → ユーザーの指摘で発覚

**根本原因**:
- Claude Code はセッションを跨ぐとコンテキストがリセットされる
- 引き継ぎサマリーに含まれる情報と、ディスク上の実ファイルを区別できなかった
- ファイルの存在確認を怠ったまま作業に着手した

**影響**: develop / main への影響なし（未コミットだったため）

**ノート**: Claude Codeは非常に有能だが、「指示されたことをすぐ実行する」特性がある。人間であれば「あれ、このファイル前にもやらなかったっけ？」と疑問を持つところを、疑わずに実行してしまった。このインシデントがプロセス改善の転機になった。

---

## スライド 10: 課題への対策 — ガードレールの設計

**対策 1: /preflight-check スキルの新設**
- Sprint 開始前に自動実行するチェックリスト
- 指示書ファイルの実在確認 → ブランチ状態確認 → 前Sprint重複チェック
- 1つでもNG → 作業開始しない

**対策 2: /session-wrapup の拡充**
- セッション終了時に「引き継ぎ注意事項」を明記
- 参照ファイルの実在確認結果、ブランチ状態を記録

**対策 3: settings.local.json による権限制限**
- 許可する操作（git, npm, gh等）を明示的にホワイトリスト化
- 不用意なシステム操作を防止

**教訓**: AIエージェントの「ガードレール」は人間が設計する責任がある

**ノート**: インシデントを受けて、問題の記録→振り返り→スキル/ルール追加→再発防止という改善サイクルを回した。これはソフトウェア開発におけるポストモーテムの手法をAIエージェント運用に適用したもの。

## スライド 11: 構成管理の仕組み — MD ファイル体系

**CLAUDE.md（10KB）** — 「AIの長期記憶」
- プロジェクトの全体像を集約（技術スタック・API一覧・デザインシステム・規約）
- セッション開始時にAIが最初に読むファイル
- セッション終了時に更新し、常に最新状態を維持

**エージェントロール MD（6ファイル）**
- 各ロール（PdO / Designer / FE / BE / QA / UX）の振る舞いガイド
- 判断基準・成功指標・担当領域を定義

**セッションログ（日次）**
- 実装内容・意思決定・残タスク・引き継ぎ注意事項を構造化

**調査ドキュメント（適宜）**
- i18n、マネタイズ、モバイル最適化等のリサーチ結果

**ノート**: CLAUDE.md は人間にとっての README であると同時に、AIにとっての「プロジェクトの記憶」として機能する。セッションを跨いでもコンテキストを失わないための中核的な仕組み。

---

## スライド 12: 構成管理の仕組み — Skills（スラッシュコマンド）

**Skills = 繰り返し実行する開発プロセスの定型化**

| コマンド | 機能 |
|---------|------|
| `/preflight-check` | 作業開始前の安全チェック |
| `/session-wrapup` | セッション終了時のドキュメント同期 |
| `/save-session-log` | 作業記録の Git 保存 |
| `/plan-issues` | 企画 → GitHub Issue 一括作成 |
| `/priority-review` | 4軸評価による優先度レビュー |
| `/notion-task` | 議事録 → Notion タスク自動登録 |

**仕組み**: 各スキルは処理手順・入出力・注意事項を定義した Markdown ファイル。スラッシュコマンドで呼び出すとAIが手順に従って自律実行する。

**ノート**: Skills は「人間が考えたベストプラクティスを、AIが再現可能な形でプログラム化したもの」と捉えられる。特に /preflight-check はインシデントから生まれたもので、失敗を仕組みで防ぐ好例。Skillsファイルは ~/.claude/commands/ に配置する。

## スライド 13: 今後の改善方針

**プロセス面**:
- CI/CD 導入（GitHub Actions で PR 時に自動テスト・ビルド）
- E2E テスト導入（Playwright）、カバレッジ 80% 目標
- エラー監視導入（Sentry）

**Claude Code 活用面**:
- CLAUDE.md のモジュール化（規模拡大に伴う肥大化対策）
- Skills の追加（`/deploy-check`、`/code-review`）
- 1セッション = 1Sprint の運用ルール確立

**プロダクト面**:
- Stripe 決済連携（本番課金）
- ダークモード対応
- 多言語対応（i18n、調査完了済み）

**ノート**: Claude Code の活用は「道具の使い方」ではなく「開発プロセスの設計」であるという認識が重要。AIの能力が上がっても、ガードレール・記録・チェックの仕組みは引き続き人間が設計・改善していく必要がある。

## スライド 14: 結論

**検証結果**:
1. **開発生産性**: 従来比 5〜10 倍の速度で同等品質のコードを生成可能
2. **品質管理**: CLAUDE.md + Skills + ブランチルールによるガードレール設計が不可欠
3. **役割の変化**: エンジニアの役割が「コード記述」から「方針決定・プロセス設計・品質管理」にシフト
4. **ドキュメント駆動**: MD ファイルがAI・人間双方の共通認識基盤として機能

**結論**:
Claude Code は「超高速な実行者」であり、判断者は人間である。AIとの協働開発を成立させるには、**AIに任せる部分と人間が設計する部分の境界を明確にし、その境界を仕組みとして実装すること**が成功の鍵である。

**ノート**: 約1週間で18,000行超のプロダクトを開発できたことは事実だが、それ以上に重要なのは「AIとの協働開発プロセスをどう設計するか」という方法論を実践的に構築できたこと。このプロセス自体が、今後の他プロジェクトにも転用可能な資産になる。
